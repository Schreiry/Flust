# FLUST - История разработки

### Аудит обновлений: от Fluminum до инженерной платформы

> *Это не changelog и не документация. Это дневник. Хроника принятых решений, найденных ошибок, внезапных озарений и ночных правок. Написана так, как она происходила - с той же долей тревоги и любопытства, с которой пишется любой настоящий программный проект.*

## Пролог. Откуда всё началось

Была программа на C++. Называлась **Fluminum**.

Она умела одно - перемножать матрицы. Но делала это с хирургической точностью: алгоритм Штрассена, ручная реализация пула потоков, AVX2 SIMD, блочное умножение с автоматически подбираемым размером тайла. Производительность измерялась в GFLOPS, результаты логировались в CSV, а монитор производительности запускался отдельным консольным процессом через Windows API.

Для своего масштаба - это был серьёзный проект. Конкретный, честный, работающий.

Но C++ - это ручное управление памятью, это Windows PDH API, который мёртв за пределами Windows, это `std::thread` с атомарными счётчиками вместо нормального work-stealing планировщика. Всё это работало, но требовало постоянного надзора.

Пришло время переписать. На Rust. И не просто переписать - переосмыслить.

Так родился **Flust**.

## Глава 0 : Контекст и конституция

До первой строчки кода было решено главное: **архитектура не обсуждается в процессе работы**. Она принимается заранее и становится законом.

Плоская структура `src/` без вложенных модулей, прямое зеркало Fluminum на C++. Никаких `matrix/core.rs`, никаких `algorithms/strassen.rs`. Каждый `.rs` файл - полноценный домен.

Были сформулированы **Десять Законов**,  не рекомендации, а инварианты. Нарушение любого из них - это регресс, а не прогресс:

- Матрица - всегда плоский `Vec<f64>`. `Vec<Vec<f64>>` - катастрофа кэша, запрещён навсегда.
- `unsafe` - только в `simd_core.rs`, только для SIMD. Перед каждым блоком - `assert!` на границах.
- Параллелизм - только `rayon`. `std::thread` - только для фонового монитора.
- `clone()` в горячем пути - преступление против производительности.
- Порядок циклов - всегда i-k-j. Порядок i-j-k - гарантированные cache misses на матрице B.
- Одна причина - один патч. Без "попробуем так".
- Сначала анализ, потом код. *Почему* это оптимально для железа - обязательный комментарий.
- `cargo check` до перехода к следующей главе.
- Тесты - истина. Ground truth - наивное умножение, epsilon = 1e-9.
- Не забегать вперёд. Если глава N не завершена - глава N+1 не начинается.

Эти законы написаны не для ИИ. Они написаны для дисциплины - чтобы проект не расползался, не накапливал долг, не превращался в груду умных идей без фундамента.

## Глава 1 - Фундамент: Cargo.toml, common.rs, matrix.rs

Первая и самая скучная глава. Именно поэтому она важнейшая.

`Cargo.toml` - это не список зависимостей. Это список инженерных решений, каждое из которых объяснено в комментарии:

- `rayon` - потому что work-stealing планировщик умнее самописного ThreadPool. `rayon::join` идеально укладывается на рекурсию Штрассена.
- `sysinfo` - потому что Windows PDH API не работает на Linux. Кроссплатформенная замена с двойным `refresh_cpu` (иначе метрика CPU всегда возвращает 0%).
- `ratatui` + `crossterm` - потому что в C++ монитор был отдельным консольным процессом через `CREATE_NEW_CONSOLE`. Теперь это поток внутри того же процесса с TUI через `alternate_screen`. Никаких висящих дочерних процессов.
- `mimalloc` (добавлен позже) - потому что `HeapAlloc` на Windows имеет глобальный lock при аллокации > 512KB, что убивает параллельность при 24 потоках.

`common.rs` стал конституцией типов. Здесь `SimdLevel`, `MultiplicationResult`, `ComparisonResult`, макрос `flust_time!`. Принцип: всё, что используется в двух и более модулях - здесь.

`matrix.rs` - главный архитектурный выбор проекта, реализованный как структура данных. `Matrix` - это:
```
rows: usize
cols: usize
data: Vec<f64>   // element(i,j) = data[i * cols + j]
```

Один непрерывный массив. Когда CPU обходит строку i, все `data[i*cols..i*cols+cols]` лежат последовательно - prefetcher загружает их в кэш-линии заранее. AVX2 грузит 4 `f64` за раз из непрерывной памяти. `Vec<Vec<f64>>` = каждая строка в произвольном месте кучи = cache miss на каждой строке.

Методы `split_4`, `combine_4`, `pad_to_power_of_2`, `unpad` - это строительные блоки алгоритма Штрассена. Написаны и покрыты тестами здесь, до того как сам Штрассен написан в главе 2.

**Критерий завершения:** `cargo check` → 0 ошибок. `test_pad_unpad_roundtrip`, `test_split_combine_roundtrip`, `test_overflow_protection` - все зелёные.

## Глава 2 - Математика: алгоритмы

Три алгоритма, три уровня сложности, один порядок написания.

**Наивное умножение** пишется первым - не потому что оно нужно, а потому что оно является ground truth. Любой другой алгоритм в любой другой момент проверяется против него. Порядок i-k-j - не рекомендация, а единственно правильный способ. Для элемента `C[i][j]` внутренний цикл по k обходит строку i матрицы A (последовательно, cache-friendly) и столбец k матрицы B (прыжки через `n` элементов). Это уже не идеально для B - именно поэтому следующий шаг это блочное умножение.

**Тайловое умножение** делит матрицы на блоки `T×T` и обрабатывает их так, что три тайла (A, B и C) одновременно помещаются в L1/L2 кэш. Размер T подбирается функцией `auto_tune_tile_size()` - она бенчмаркирует семь вариантов `{16, 24, 32, 48, 64, 96, 128}` на матрицах 256×256 (3 прогона, берётся минимум - для исключения GC-пауз), выбирает быстрейший. Это вызывается один раз при старте и кэшируется в `AppState`.

**Алгоритм Штрассена** (Volker Strassen, 1969, *"Gaussian Elimination is not Optimal"*) - центральный вычислительный алгоритм проекта. Вместо 8 рекурсивных умножений подматриц - только 7, за счёт алгебраической перестройки формул. Это снижает сложность с O(n³) до O(n²·⁸⁰⁷).

Перед написанием Штрассена была нарисована таблица использования квадрантов - сколько раз каждый квадрант участвует в вычислении M₁...M₇. Это единственный способ понять, где нужен `clone()` (квадрант используется больше одного раза), а где достаточно owned-значения. Бесконтрольный `clone()` в рекурсии - это "clone storm": при n=4096 и threshold=64 около шести уровней рекурсии, каждый из которых без дисциплины взрывает потребление RAM.

Условие завершения главы - `test_strassen_matches_naive_255x255`. Именно несимметричный размер, именно с padding/unpadding. Если это проходит - Штрассен правильный.

## Глава 3 - Железо: system.rs + simd_core.rs

Самая опасная глава. Здесь живёт весь `unsafe` код проекта.

`system.rs` делает три вещи: определяет уровень SIMD через `is_x86_feature_detected!` (AVX-512, AVX2, SSE4.2 или scalar fallback), собирает `SystemInfo` через `sysinfo` (RAM, CPU, количество ядер), запускает `auto_tune_tile_size()`.

`simd_core.rs` - изолированный модуль с `#[target_feature(enable = "avx2,fma")]`. Принцип: безопасность проверяется **снаружи** `unsafe` блока. Внутри - только математика, никаких `if/else`. Горячий AVX2-цикл для каждой строки i и каждого элемента k:

```rust
let a_ik = _mm256_set1_pd(*a.get_unchecked(i * n + k));
// шаг j = 4:
let b_chunk = _mm256_loadu_pd(b_ptr);
let c_chunk = _mm256_loadu_pd(c_ptr as *const f64);
let result  = _mm256_fmadd_pd(a_ik, b_chunk, c_chunk);  // FMA
_mm256_storeu_pd(c_ptr, result);
```

`_mm256_fmadd_pd` - это fused multiply-add: `a_ik * b_chunk + c_chunk` за одну инструкцию. Отдельные multiply и add дали бы два обращения к исполнительному устройству и потенциальную потерю точности из-за промежуточного округления. FMA - правильный выбор.

Диспетчер `multiply_dispatch` переключается между AVX-512, AVX2, SSE4.2 и scalar **в runtime**. Бинарник остаётся переносимым - определение SIMD-уровня происходит при каждом запуске.

## Глава 4 - Монитор производительности

В Fluminum монитор был отдельным процессом через `CREATE_NEW_CONSOLE`. Это Windows-only, нет shared memory, нужен IPC, могут виснуть дочерние процессы.

В Flust - это `std::thread` внутри того же процесса. Метрики собирает `MetricsCollector` в фоновом потоке, данные передаются через `Arc<Mutex<MonitorHistory>>`. Рендеринг - через `ratatui` в `alternate_screen`, что не затрагивает основной вывод.

Один критичный нюанс с `sysinfo`: CPU usage - это **дельта** между двумя вызовами `refresh_cpu`. Нельзя вызывать их без паузы - вернётся 0%. В MetricsCollector первый refresh делается в конструкторе (с задержкой 200ms), все последующие - в `collect()` при каждом опросе (каждые 250ms). Эта тонкость стоила отдельного `TODO` в коде и потом была исправлена явно.

RAII-guard `TerminalGuard` гарантирует восстановление терминала даже при панике. `impl Drop for TerminalGuard { fn drop(&mut self) { disable_raw_mode(); LeaveAlternateScreen; } }` - маленькая деталь, которая стоит часов отладки "сломанного" консольного состояния при разработке.

## Глава 5 - Интерфейс: io.rs + interactive.rs

`io.rs` - это словарь UI. ANSI-константы, box-drawing функции, типобезопасный `prompt<T>()`, CSV-логирование через `serde`, файловый ввод/вывод матриц в формате Fluminum (первая строка `rows,cols`, потом данные). Совместимость с Fluminum-файлами была принципиальной: история проекта не должна обрываться.

`interactive.rs` стал главным местом жизни пользователя. `AppState` инициализируется один раз при старте: collect SystemInfo → auto_tune → detect SIMD → show progress. Потом - основной цикл меню.

## Глава 18 - Критические исправления

Правило главы: каждый шаг - отдельная ветка изменений. `cargo check` после каждого. Если шаг ухудшает - откат немедленно.

**Регрессия производительности** (~70% замедление Winograd). Диагностика в правильном порядке:

1. Clone storm в Winograd? → Закомментировать, прогнать только Strassen. Если Strassen тоже замедлился - проблема не здесь.
2. `Arc<Mutex<>>` contention в hot-path? → `grep -rn "lock()" src/algorithms.rs src/matrix.rs`. В алгоритмах не должно быть ни одного `lock()`.
3. Debug build вместо release? → Разница 10–50× для матричных операций.
4. Лишние аллокации `ThemeColors::new()` в render loop?
5. `auto_tune_tile_size` запускается не один раз?

Нашли причину, устранили - `bench_quick()` вернул ~350ms.

**Тема не применяется к монитору.** Корневая причина: `run_monitor_tui()` запускается в отдельном потоке до того, как пользователь успевает сменить тему. Решение: `Arc<RwLock<ThemeColors>>` вместо копии темы при запуске. Монитор держит дескриптор, читает текущую тему при каждом кадре.

**Навигация в меню алгоритмов.** Индекс курсора не опускался ниже "Hybrid Full Core". Баг в логике границ массива - классическое off-by-one при расчёте `max_index`. Пункт "Intel MKL" был отрисован, но недостижим по той же причине. Пункт "Back" удалён из рендеринга и логики.

## Глава 19 - Тепловая симуляция: помочь другу

Здесь история становится личной.

Есть друг - Александр. У него инженерный проект: робот-танк, который ищет источник тепла через инфракрасный сенсор. Тепло накапливается в резервуаре с жидкостью, термоэлектрический генератор (ТЭГ) преобразует температурный перепад в электрическую энергию для моторов. Вопрос: сколько времени проработают моторы до того, как жидкость остынет настолько, что напряжение ТЭГ упадёт ниже рабочего минимума?

Это не абстрактная задача. Это конкретный физический объект, который нужно рассчитать.

Модуль `thermal.rs` реализует **3D решатель теплопроводности методом конечных разностей (FDM)** на базе уравнения Фурье:

```
∂T/∂t = α · (∂²T/∂x² + ∂²T/∂y² + ∂²T/∂z²)
```

где `α = λ / (ρ·c)` - коэффициент температуропроводности. Сетка Nₓ×N_y×N_z, линейный индекс узла: `idx = i*Ny*Nz + j*Nz + k`.

**Матричная форма** (явная схема Эйлера):
```
T_new = A · T_old,   где  A = I + dt·α·L
```
где `L` - разреженная матрица Лапласа, построенная в CSR-формате. Это не абстракция - это конкретная разреженная матрица `(N·N·N) × (N·N·N)` с семью ненулевыми диагоналями (центр + 6 соседей по осям).

**Условие устойчивости Куранта** проверяется до запуска симуляции:
```
dt ≤ 1 / (2·α · (1/hₓ² + 1/h_y² + 1/h_z²))
```
Нарушение - численная нестабильность, температура уходит в ±∞. Программа отклоняет конфигурацию с понятным сообщением.

**Граничные условия Дирихле:** граничные узлы фиксированы при `T_boundary` (температура окружающей среды). В матрице `A` строка граничного узла - единичная строка (диагональ = 1, остальное = 0).

**Эффект Зеебека** (T.J. Seebeck, 1821):
```
V = S · ΔT          (напряжение ТЭГ)
I = V / (R_int + R_load)
P = I² · R_load     (мощность)
```
Для Bi₂Te₃ (реально доступный для студенческого проекта): `S ≈ 0.05 В/К`. Максимальная мощность при `R_load = R_internal` - классическое согласование нагрузки.

Пресет `config_tank_project_default()` настроен на реалистичные параметры: резервуар 150×80×60 мм, жидкость (вода) при 85°C, окружающая среда 20°C, сетка 16×16×16 (4096 узлов, ~100ms), ТЭГ на боковой стенке.

Симуляция при этих параметрах даёт пиковое напряжение ≈ 3.1 В (при ΔT ≈ 62°C), время работы моторов ≈ 12–18 минут.

Экспорт в CSV - полный: метаданные в `#`-комментариях, данные в виде `time_s,step,t_center_C,voltage_V,power_mW,...`. Совместим с Python/pandas, MATLAB, Excel. Визуализация в терминале: тепловой срез через ASCII/Braille - горячий центр как `▓▓▓`, холодные края как `···`.

## Глава 20 - VTune: диагноз произнесён

Intel VTune Amplifier. i9-13900K, HT отключён, 24 ядра / 24 потока.

Цифры были неутешительными:

```
Logical Core Utilization:  9.7%  (работает 2.3 ядра из 24)
Back-End Bound:           58.4%
Memory Bound:             38.0%
L3 Bound:                 14.8%
IPC:                      0.696  (норма для AVX matmul: 2.5–4.0)
DP GFLOPS:                1.498  (потенциал при 24 ядрах AVX2: 200+ GFLOPS)
```

Диагноз в одном предложении: **Rayon запускает 7 задач Strassen первого уровня - и 17 ядер простаивают**, пока эти 7 выполняются.

Детали для i9-13900K:

- Матрица 2048×2048, threshold=64 → уровень 1: 7 задач (1024×1024 каждая) → 7 из 24 ядер заняты, остальные ждут.
- Матрица 4096×4096, threshold=64 → 5 уровней → огромный простой в начале.
- Тайлы 64×64 не помещаются в L1 (48 KB для P-core): `3 × 64² × 8 = 98 304 байта > 48 KB` → данные вытесняются в L2/L3 → L3 Bound 14.8%.

**Первое исправление - снизить threshold:**
```
Было:  STRASSEN_THRESHOLD = 64  → 4 уровня для n=1024
Стало: STRASSEN_THRESHOLD = 32  → 5 уровней для n=1024
```
На уровне 2 при threshold=32: 49 задач (512×512) → все 24 ядра заняты уже на втором уровне.

**Второе исправление - outer-parallel Strassen:** перед рекурсией матрица делится на K горизонтальных полос (K = `rayon::current_num_threads()`). Каждый поток получает независимый слайс с первого шага - не нужно ждать рекурсии.

**Третье исправление - двухуровневый тайлинг:**
```
Outer tile T_L2 = 256  (для L2 кэша P-core 2 MB)
Inner tile T_L1 = 32   (для L1 кэша P-core 48 KB)
```
Параллелизация только по внешним i-блокам T_L2.

## Глава 21 - Глубокая оптимизация

Три категории проблем. Все три решаются в одной главе.

**Аллокатор памяти.** Windows `HeapAlloc` имеет глобальный lock при аллокации > 512 KB. Штрассен при 24 параллельных потоках создаёт матрицы 1024×1024 (8 MB каждая) - все потоки выстраиваются в очередь к одному локy. Решение: `mimalloc` как глобальный аллокатор.

```rust
use mimalloc::MiMalloc;
#[global_allocator]
static GLOBAL: MiMalloc = MiMalloc;
```

Одна строка в `main.rs`. 20–40% прирост на параллельных аллокациях.

**Выровненная память.** AVX2 `_mm256_loadu_pd` работает с невыровненными адресами - но медленнее. `_mm256_load_pd` требует 32-байтного выравнивания. Кастомная аллокация через `std::alloc::Layout::from_size_align(n * 8, 32)` или крейт `aligned-vec` дают выровненный `Vec<f64>`.

**Rayon ThreadPool под гибридную архитектуру.** i9-13900K - P+E гибрид: 8 P-cores (Performance, ~5.4 GHz, 16 логических с HT) + 8 E-cores (Efficient, ~4.0 GHz, 8 логических). При AVX2 workload hyperthreading на P-cores не помогает: оба логических потока делят AVX-исполнительный блок. Оптимум для матричного умножения:

```rust
rayon::ThreadPoolBuilder::new()
    .num_threads(physical_cores)   // 16 P-cores или 24 при HT-off
    .stack_size(4 * 1024 * 1024)
    .build_global().unwrap();
```

**LLVM-флаги** в `.cargo/config.toml`:
```
"-C", "target-feature=+avx2,+fma,+bmi1,+bmi2,+popcnt"
"-C", "llvm-args=-unroll-count=4"   // размотка внутреннего j-цикла
```

## Что есть сейчас

Flust - это уже не просто порт Fluminum. Это несколько связанных вещей одновременно:

**Вычислительный движок.** Алгоритм Штрассена с outer-parallel декомпозицией, двухуровневым тайлингом, AVX2/FMA SIMD ядром. Алгоритм Винограда. Авто-тюнер размера тайла. Измерение GFLOPS с разбивкой по фазам (padding, compute, unpadding). Экспорт результатов в CSV.

**Инженерный инструмент.** 3D FDM решатель теплопроводности с полноценной физикой (уравнение Фурье, условие Куранта, граничные условия Дирихле, эффект Зеебека). Спроектированный не в вакууме, а для конкретной задачи - помочь другу рассчитать проект.

**Аппаратная лаборатория.** Performance Monitor с per-core барами для 24 ядер, sparkline историей загрузки, gauge памяти. VTune-диагностика с конкретными числами и конкретным планом исправлений. Место, где можно видеть что происходит внутри железа.

**TUI-платформа.** Тема, конфигурация, История результатов, System Information, Benchmark Suite. Не консоль с цифрами - полноценный терминальный интерфейс.

## Что будет дальше

Есть несколько направлений, которые лежат перед проектом.

**Engineering Menu** - раздел с инженерными задачами. В нём:

*Kinematics & Pathfinding (MDP).* Симуляция движения в неизвестной среде - марковские процессы принятия решений. Матрица переходов P, возведение в степень до сходимости (Binary Matrix Exponentiation). Матрицы состояний до 2048×2048. Именно здесь HPC-ядро Flust показывает свою универсальность - та же операция умножения, другой физический смысл.

*Computer Vision (IR Processing).* Обработка сырых данных с ИК-матрицы: интерполяция (бикубическая), свёрточная фильтрация. Свёртка разворачивается в GEMM через преобразование `im2col`. Входные данные - матрица 8×8 пикселей (AMG8833), выход - интерполированная матрица высокого разрешения.

**Разреженные матрицы** (CSR/CSC) - открывают FEA/CFD задачи. Уже частично реализованы в `sparse.rs` для тепловой симуляции.

**Линейные решатели** - LU-разложение, метод сопряжённых градиентов, QR-факторизация. Для решения систем `Ax = b`.

**Eigensolvers** - Power Method (PageRank), QR Algorithm, алгоритм Ланцоша для больших разреженных матриц.

## Эпилог

Когда смотришь на Flust со стороны - видишь список алгоритмов и файлов. Когда смотришь изнутри - видишь последовательность решений, каждое из которых было принято по конкретной причине.

Матрица как плоский вектор - потому что кэш-линии.  
Rayon вместо threads - потому что work-stealing умнее статического распределения задач.  
Unsafe только в SIMD - потому что unsafe без причины - это долг, который платишь багами.  
Десять законов - потому что дисциплина в одиночку определяет качество продукта, когда нет команды с code review.

И тепловая симуляция - потому что у Александра есть проект, и ему нужна была программа, которая посчитает, сколько минут проработает его танк.

Это достаточная причина писать хороший код.
